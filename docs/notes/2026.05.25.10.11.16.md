 (世, p=0.9) What is happening:

 1. A Discord-triggered run completed or failed.
 2. openplanner-memory/index-run-memory! tried to archive/index that finished run.
 3. Because session-store* exists, it delegated to the new composite session store:
   index-run-memory! → complete-run! store (:run_id run) ...
 4. The composite store first asks Redis to complete the run.
 5. Redis implements completion as patch-run!.
 6. Redis could not find the run key, so it threw:
   patch-run! on unknown run
 7. The OpenPlanner memory layer catches that broadly and logs:
   run indexing failed; continuing without OpenPlanner persistence

 So the actual failure is probably not an OpenPlanner service/indexing outage. It is likely a local persistence lifecycle
 mismatch: the finalizer is trying to complete a run in the new ISessionStore, but the initial run was never written
 there.

 The suspicious path is:

 - create-initial-run! stores the run in the in-memory runs* atom via store-run!.
 - It also writes legacy session state via session-store/put-session!.
 - It does not appear to write the initial base-run through @session-store* / put-run!.
 - Later, index-run-memory! assumes the run exists in @session-store* and calls complete-run!.

 That explains “unknown run.”

 So there may be a second issue: Discord jobs can launch without a stable job id, producing low-quality run identifiers.

 (己, p=0.88) Frame 1: This is a migration seam bug. The code is halfway between legacy run/session persistence and the
 newer ISessionStore abstraction. Creation and completion are not using the same authority.

 (己, p=0.74) Frame 2: This could also happen after Redis TTL expiry, Redis restart, or a long-running job exceeding

 (己, p=0.8) Frame 3: The error handling is fail-open in the right broad sense — a user turn should not fail just because

We pick Option 1

Every session store call should be calling the composite session store.
No session dependencies should be interacting  directly with  redis or openplanner, only through a composite session store.

redis is primary during a live streaming session
