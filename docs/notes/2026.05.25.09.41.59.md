 (己, p=0.96) Yes — you’re asking exactly the right questions. infer-externs only really matters when the Closure compiler might rename/minify JavaScript property names, which mainly means :advanced optimization builds. If Knoxx backend builds are
 :optimizations :none or :simple, disabling :infer-externs is usually reasonable because there is no advanced property-renaming pass for it to protect.

 Short version:

 ```clojure
   :optimizations :advanced
   ;; Closure aggressively renames code.
   ;; JS interop like (.-foo x) can break unless "foo" is protected.
   ;; externs tell Closure: do not rename these external JS properties.

   :infer-externs :auto
   ;; Shadow tries to infer those externs from your JS interop.
   ;; Helpful in theory.
   ;; Noisy/frustrating in dynamic JS-heavy CLJS code.

   :optimizations :none / :simple
   ;; Closure is not doing the same destructive property renaming.
   ;; Extern inference warnings are often just noise.
   ;; infer-externs=false can be fine.
 ```

 So when I said “non-advanced,” I meant builds that are not using:

 ```clojure
   :optimizations :advanced
 ```

 Knoxx backend currently has:

 ```clojure
   :optimizations :none
 ```

 for :server, :server-dev, :test-ci, and:

 ```clojure
   :optimizations :simple
 ```

 for :app.

 Those are “non-advanced” in this context.

 Evidence

 (世, p=0.94) ClojureScript compiles through Google Closure Compiler. Closure has optimization levels. In :advanced, Closure assumes it can rewrite/rename aggressively to shrink and optimize output.

 Example problem:

 ```clojure
   (defn username [user]
     (.-username user))
 ```

 In advanced mode, Closure might rename internal properties. But user might be a raw object from Discord.js, Fastify, Node, Redis, etc. Closure does not own those libraries’ runtime object shapes. If Closure thinks .username is safe to rename,
 emitted JS can stop asking for the actual external property named "username".

 Externs are declarations that say, approximately:

 ```js
   /** @externs */
   var DiscordUser = {};
   DiscordUser.prototype.username;
 ```

 Meaning:

 │ This property exists outside compiled CLJS. Do not rename it.

 (己, p=0.9) infer-externs is Shadow CLJS trying to generate those extern declarations automatically from code patterns like:

 ```clojure
   (.-username user)
   (.send reply payload)
   (.fetch client id)
 ```

 But it needs to know whether user, reply, client, etc. are JS objects. If it cannot infer the target type, it emits warnings like:

 ```text
   Cannot infer target type in expression (. reply send ...)
 ```

 Usually the workaround is to annotate:

 ```clojure
   (fn [^js reply]
     (.send reply payload))
 ```

 or:

 ```clojure
   (let [^js raw-res (aget reply "raw")]
     (.writeHead raw-res 500 headers))
 ```

 (己, p=0.87) That is why these warnings felt endless in Knoxx: backend CLJS talks to many dynamic JS boundaries — Fastify, Node streams, Redis clients, Discord.js, MCP SDK, file stats, raw HTTP responses — and many values are intentionally opaque
 JS handles. Shadow can’t always infer them.

 Frames

 (汝, p=0.86) One interpretation: your intuition that “it never seemed able to work” is valid for this style of code. infer-externs works better when JS interop is narrow, typed, and consistently hinted; it works poorly when raw JS objects flow
 through many dynamic route handlers.

 (己, p=0.82) Another interpretation: infer-externs is not “broken”; it is solving a problem we mostly do not have in backend :none/:simple builds, while creating warning noise that hides real defects like undeclared vars and wrong arity.

 (世, p=0.72) A third interpretation: if Knoxx ever adds an actual browser/frontend :advanced build, then externs become real again. At that point we should not simply disable inference; we should either add proper ^js hints, explicit extern files,
 or avoid raw interop outside extern adapters.

 Countermoves

 (己, p=0.93) The dangerous misunderstanding would be: “infer-externs=false is always safe.” It is not. It is safe-ish only when the build is not doing advanced Closure optimization, or when externs are handled another way.

 (己, p=0.9) The other dangerous misunderstanding would be: “all infer warnings are fake.” Also not true. Sometimes infer warnings point to a boundary leak: a raw JS object is being touched outside an extern adapter, or a value should be hinted as
 ^js.

 (己, p=0.84) For Knoxx, the better policy is:

 ```clojure
   ;; non-advanced backend/server/test builds
   :infer-externs false

   ;; advanced browser/library builds
   :infer-externs :auto
   ;; plus explicit ^js hints or extern adapter wrappers
 ```

 │ Is this build using Closure advanced optimizations, and therefore does it need extern protection?

 For this backend path, the answer is currently no.
