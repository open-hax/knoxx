
 (世, p=0.95) specs/epics/events-agent-runtime-separation.md explicitly says
 the unwanted concept is event-agent job, because it mixes:

 - trigger definition
 - source subscription
 - dispatch action
 - agent template

 (世, p=0.94) docs/notes/architecture/event-trigger-action-runtime.md says:

 - agents do not carry source, trigger, or event trigger fields
 - trigger contracts define when work starts
 - actions are registry functions invoked against actor context
 - cron/schedule should emit events then use the same trigger/action path

 (世, p=0.92) docs/notes/architecture/action-registry-intent.md gives the
 cleaner vocabulary:

 - Actor = persistent entity with identity/state/policy
 - Generator = entity with policy that produces events
 - Schedule = temporal rule that emits synthetic events, implementing
 generator shape
 - Trigger = agreement to take action in response to event + condition
 - Agent = driver/runner/executable
 - Tool = action bundled with usage text

 docs/notes/architecture/data-oriented-patterns.md,  supports the same
 constraints via tables of records, data + interpreter, event log +
 projections, schema at edge, and command-as-data.
