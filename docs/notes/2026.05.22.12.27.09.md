  Boundary violation map

  is to invert it: callers import and call the CLJS public functions directly.

  ---
  What the clean shape looks like

  create-policy-db returns a CLJS map:
  {:pool pool :primary-org primary-org :bootstrap bootstrap}

  or just pool — callers that need primary-org or bootstrap get them threaded
  in at startup. All callers replace .method(...) with a direct ns-qualified
  call:

  ;; before (boundary violation)
  (.resolveRequestContext policyDb headers)

  ;; after (CLJS boundary respected)
  (policy-db/resolve-request-context! pool headers)

  extern/pg.cljs is the only file that ever sees a JS pool object or calls
  .query. infra/db/policy.cljs receives it as an opaque value and passes it to
  pg/query! — that's the contract.

  The build-facade / ctx->js / session->js / bootstrap->js functions I added
  should be removed once the callers are fixed. They exist right now only to
  unblock login.

 The scope:
  - create-policy-db returns CLJS map {:pool :primary-org :bootstrap}
  - Remove the entire facade section
  - Thread pool/context properly through every caller
  - Routes convert HTTP bodies at the boundary, call CLJS fns directly
