(己, p=0.92)
backend/src/cljs/knoxx/backend/infra/control_config.cljs is currently a
mixed-concern adapter, and the name “control config” hides at least four
separate jobs.

(己, p=0.9) What this file appears to be:

1. Legacy runtime control-state assembly
- Loads/saves “event-agent-control” overrides from Redis.
- Builds the persisted/default control map consumed elsewhere.
2. Contract-to-runtime projection
- Reads agent contracts.
- Resolves roles/tools/prompts/source modes.
- Projects those contracts into runnable job maps.
3. Shape construction / normalization
- Builds nested maps like :trigger, :source, :agentSpec, :filters.
- Clamps message limits.
- Normalizes tool policies, channels, keywords, source config.
4. Discord-specific defaults
- Reads Discord env vars.
- Defines default Discord jobs.
- Provides Discord channels, keywords, tool policies, model fallback.

(己, p=0.95) That violates the local architecture principle from
docs/notes/architecture/data-oriented-patterns.md: this namespace is not just
“data + interpreter”; it is simultaneously schema-ish shape definition,
legacy persistence, Discord defaults, and contract projection.

(己, p=0.87) The most suspicious pieces are the “event-agent” surface names.
If event-agents were a bad/removed concept, this file is probably a
compatibility fossil. I would not preserve that name in new code.

Proposed split:

```text
backend/src/cljs/knoxx/backend/shape/control.cljs
Malli/data shapes for control jobs, triggers, sources, agent specs.

backend/src/cljs/knoxx/backend/law/control.cljs
Pure invariants:
- clamp max messages
- normalize trigger kind
- normalize source kind
- normalize tool policy
     - validate source config semantics

   backend/src/cljs/knoxx/backend/domain/control/contracts.cljs
     Pure contract -> control/job projection.
     No Redis. No env. No Discord env reads.

   backend/src/cljs/knoxx/backend/domain/discord/control.cljs
     Discord defaults:
     - default channels
     - default keywords
     - default Discord tool policies
     - legacy Discord job templates, if still needed

   backend/src/cljs/knoxx/backend/infra/control_config.cljs
     Thin shell only:
     - Redis load/save
     - env access
     - calls pure domain/law/shape functions
 ```

(己, p=0.84) Naming recommendation: replace public “event-agent” naming with

 ```clojure
   control-job

 ```


(世, p=0.88) docs/notes/architecture/data-oriented-patterns.md says the
principles are:

- Table of Records
- Data + Interpreter
- Pipeline of Transforms
- Schema at the Edge
- Strategy as Data
- Command as Plain Data, Handler as Pure Function

This file only partially follows those. It emits plain maps, but the
transformation boundaries are muddy.

Frames

(己, p=0.87) Frame 1: “Compatibility adapter that grew barnacles.”
The namespace may once have been reasonable as a Redis-backed config loader,
then accumulated contract projection, Discord defaults, and migration
support.

(己, p=0.82) Frame 2: “Old runtime concept leaking into the new
architecture.”
The event-agent terminology probably persisted because API/UI/persistence
keys depended on it, even after the actual conceptual model changed.

(己, p=0.78) Frame 3: “Shape definitions hiding inside constructors.”
Functions like contract-source-spec, contract-agent-spec, and
normalize-event-agent-job are effectively defining the runtime data model by
example, rather than declaring shapes in shape/ and enforcing laws in law/.

Countermoves

(己, p=0.9) Do not delete all event-agent names blindly. First inspect
callers and persisted Redis keys; the name may still be part of a
compatibility API even if the concept is dead.

(己, p=0.88) Do not move Redis into domain code. Redis belongs in infra; the
projection from contracts to job maps should be pure.

(己, p=0.86) Do not turn this into OO-style builders. The right direction is
still plain maps plus pure transform functions; just put shapes, laws, and
adapters in the right namespaces.

(己, p=0.8) Keep legacy aliases if frontend/API clients still consume
event-agent-control-config; add new names first, then deprecate old public
entry points.
