## Discord keyword

A driver can't emit a discord.message.keyword event.
that'd require the driver implementation to know about the keywords
but it doesn't, it can't, it shouldn't

we can later create an action that dispatches such an event in response to a trigger that fires on a message event.
but we can't expect the driver to emit this.

## Driver Protocol
And we definitely shouldn't be hard coding the driver specs into the registry.
what makes a driver has to be abstract enough as a pattern to reuse anywhere.

This driver protocol is meant for driver, say discord, to implement.

so we implement a discord driver record, that the discord gateway implements, so the gateway can dispatch events into the system easily.

so our source resources can specificly choose which events to register the resource for

And why is the driver conccept baked into the registry?

come on, seperate the driver concept as a domain.
